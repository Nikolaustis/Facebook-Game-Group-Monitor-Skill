# V6.2.2 Validation

## Required behavior

1. No shutdown wording -> generated policy mode is `none`.
2. Explicit no shutdown -> `none`, even if task config contains legacy shutdown fields.
3. Explicit shutdown after completion -> `after_complete`.
4. Conditional shutdown -> `before_deadline` with timezone-aware deadline.
5. Missing deadline for `before_deadline` is rejected before task registration.
6. Scheduled runner and direct fallback pass the same `shutdown_policy.json` to phase 2.
7. Node treats policy mode `none` as authoritative and ignores stale config shutdown settings.
8. Policy JSON with a UTF-8 BOM is accepted.
9. Shutdown watcher waits for the PowerShell runner, re-cleans the deterministic main task, and the one-shot fallback deletes its own task definition before calling shutdown.
10. The policy is re-read at finalization, so a later Codex instruction can change or cancel shutdown; invalid updates fail closed to `none`.

## Static validation performed

- `node --check scripts/phase2_collect_details.js`
- `package.json`, `task_config.template.json` and `shutdown_policy.template.json` JSON parse
- Run-policy loader tests for `none`, `after_complete`, `before_deadline`, invalid mode, missing deadline and BOM
- Manifest/direct runner references use `--shutdown-policy-file`, `--shutdown-wait-pid`, and scheduled task identity
- Watcher self-deletion command is encoded into the one-shot task before `shutdown.exe`
- Documentation states default no shutdown and prohibits user file editing

Windows Task Scheduler and actual shutdown remain Windows-only integration tests and must be verified on the target machine.
