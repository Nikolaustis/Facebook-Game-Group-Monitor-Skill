# V6.6.0 Incident Analysis

## Incident

A queued phase-2 task was prepared after another Facebook collection. The handoff did start after the first task completed and Facebook login validation succeeded, but phase 2 exited immediately while parsing generated JSON.

## Direct cause

The temporary PowerShell handoff wrote `phase1_index.json` and/or `task_config.json` with Windows PowerShell 5.1 UTF-8 BOM semantics. The phase-2 entry path used raw `JSON.parse(fs.readFileSync(..., 'utf8'))`, so the encoding marker was treated as invalid JSON input.

## Contributing defects

- Entry JSON parsing was inconsistent across scripts.
- The handoff used an ad hoc writer instead of a package-level no-BOM writer.
- Launch success was inferred from process creation rather than a fresh progress checkpoint.
- The temporary waiter did not monitor immediate stderr/exit status or retry before the requested operational window elapsed.

## V6.6.0 correction

- Unified JSON decoding handles UTF-8 BOM and UTF-16.
- All target files are parsed before Chrome and scheduler launch.
- A supervisor requires a fresh readable progress checkpoint before startup is accepted.
- A standard handoff script replaces temporary wait logic and records/retries startup failure.
- Core PowerShell JSON writers use UTF-8 without BOM.

The error was triggered by temporary orchestration output, while the former Skill’s input and health-check gaps allowed it to become an unattended overnight failure. V6.6.0 addresses both layers.
