# V6.6.1 Incident Analysis

## Incident

The Codex/ChatGPT desktop application failed to start after the user-level environment variable below was created during Codex CLI troubleshooting:

```text
CODEX_CLI_PATH=C:\Users\Og\AppData\Roaming\npm\codex.cmd
```

Removing the variable and restarting restored desktop startup. The Skill did not create the variable itself, but its diagnostics previously recommended configuring it and its resolver read it with high priority.

## Root cause boundary

- Installing the standalone `@openai/codex` CLI was not the fault.
- Keeping `codex.cmd` on PATH was not the fault.
- Calling `codex exec` from the Skill was not inherently the fault.
- The conflicting global variable name was the high-confidence trigger.
- The Skill contributed indirectly by accepting and recommending that globally shared variable even though it already had PATH/npm discovery.

## V6.6.1 correction

- The resolver no longer reads `CODEX_CLI_PATH`.
- The variable is stripped from all resolver child-process environments.
- The private optional override is `FB_MONITOR_CODEX_CLI_PATH`.
- A direct CLI path belongs in `config/local/semantic_model.local.json` under `codex_exec.command`.
- Diagnostics expose only whether the legacy variable exists; its value is never written.
- Cleanup is explicit and user-triggered, never automatic.

This separates Skill configuration from environment names potentially consumed by the desktop application.
