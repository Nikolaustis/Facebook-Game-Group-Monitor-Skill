# V6.3.0 Validation

## Incident evidence addressed

- V6.2.2 completion recorded watcher PID 27420 and `conditional_forced_shutdown_powershell_watcher_started`.
- No `conditional_shutdown_watcher_status.json` was created.
- No watcher stdout/stderr or PowerShell engine startup evidence existed.
- The scheduled runner itself completed normally and deleted its task.
- Therefore the failure boundary was detached watcher startup/health confirmation, not phase2 collection, finalization, Chrome closure, policy resolution, or the power guard.

## Required behavior

1. Scheduled and direct background launchers pass `--shutdown-coordinator-mode runner`.
2. Node does not spawn a detached watcher in runner mode.
3. Node writes `completed_shutdown_pending_runner` and a non-empty random request token.
4. The existing hidden runner stops the power guard and removes the main scheduled task before invoking the coordinator.
5. The coordinator re-reads the current run-local policy and strict finalization files.
6. Policy `none`, invalid policy, failed validation, or expired deadline never shuts down.
7. Enabled valid policy calls full-path `shutdown.exe /s /f /t ...`; failures fall back to Start-Process and a self-deleting one-shot task.
8. Coordinator status, stdout and stderr are always retained for diagnostics.
9. The final completion JSON records the actual coordinator result, not merely a child PID.
10. Default behavior remains no shutdown unless the current Codex instruction explicitly enables it.
