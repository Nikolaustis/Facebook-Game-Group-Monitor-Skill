# V6.3.0 Incident Analysis: shutdown watcher never entered script

## Evidence from the Ragnarok six-game run

- Phase 2 and finalization completed successfully.
- `codex_task_complete.json` recorded `shutdown_mode=after_complete`, `phase2_finalization_verified=true`, `chrome_closed=true`, and watcher PID 27420.
- No `conditional_shutdown_watcher_status.json` was created.
- Scheduled phase2 stderr was empty.
- The runner status was `finished_success`, and the main scheduled task was deleted.
- The runtime power guard stopped normally and recorded zero successful shutdown aborts.

## Root cause boundary

V6.2.2 returned success immediately after Node `spawn()` produced a PID. It did not wait for the watcher to write its first health status, did not retain stdout/stderr, and did not verify that PowerShell entered `conditional_shutdown_watcher.ps1`. On the target machine, the detached child exited before the first script statement, so `shutdown.exe` was never invoked.

The records do not contain the detached child exit code or stderr, so they cannot prove whether the immediate exit was caused by Windows job/process containment, PowerShell startup/parameter binding, or another process-creation constraint. The exact low-level cause was made unobservable by V6.2.2 itself.

## V6.3.0 architecture

The detached watcher is removed from normal scheduled/direct background execution. The already-healthy hidden PowerShell runner now performs shutdown coordination synchronously after Node completes, the power guard stops, and the main task definition is deleted. This removes the failing process boundary and creates complete status/stdout/stderr diagnostics for every shutdown attempt.
