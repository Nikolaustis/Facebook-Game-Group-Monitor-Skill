# V6.5.4 Incident Analysis

## Incident

The API-only verification command returned a nonzero exit code when a configured API successfully returned valid Schema output below the confidence threshold. Runtime behavior was correct: the low-confidence decision continued to fallback. The diagnostic command nevertheless classified the final `rules_only` provider as an API failure.

## Root cause

The diagnostic used this overly broad condition:

```javascript
if (apiOnly && result.provider === 'rules_only') process.exitCode = 2;
```

That condition examined only the final provider. It did not distinguish:

- API request or parsing failure; from
- valid low-confidence API output followed by intentional fallback.

## Corrective action

V6.5.4 evaluates the provider chain and API statistics. It treats these outcomes separately:

```text
custom_api:* accepted → verification success
custom_api:* low_confidence → verification success with fallback status
HTTP / network / invalid JSON / invalid Schema → verification failure
no configured API or no request → verification failure
```

The diagnostic report now includes a structured `verification` object and uses it to set the process exit code.

## Runtime impact

The production semantic provider chain was already functional. This correction affects the API self-test command and its reporting, not the second-round collection logic.
